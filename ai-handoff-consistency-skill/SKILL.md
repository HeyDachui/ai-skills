# 对话一致性回验 Skill

## 1. Skill 目标

本 Skill 用于解决以下问题：

新对话表面理解了任务，也按格式输出了内容，但实际方向和用户真实意图不一致。

本 Skill 的目标不是让 AI 更会写，也不是让 AI 更快进入生产，而是让新对话在接手旧项目时，先完成：

- 理解项目边界
- 理解用户真实意图
- 理解当前阶段
- 理解不能做什么
- 理解如何接受纠偏
- 通过可回验的一致性验收

核心自检句：

未通过一致性验收前，不得进入生产。

---

## 2. 适用场景

当用户提出以下需求时，应触发本 Skill：

- 接手这个项目
- 继续上一个项目
- 这是交接文档
- 帮我开一个新对话接手
- 旧对话太长了，我要换新对话
- 请让新对话理解这个项目
- 帮我给 Codex 新线程写接手指令
- 请检查新对话是否理解正确
- 请生成一致性验收内容
- 请把这个项目交给另一个 AI 对话

本 Skill 适用于：

- 新 ChatGPT 对话接手旧项目
- Codex 新线程接手旧项目
- 项目交接
- 旧对话卡顿后的新对话接续
- 已有项目协议需要被新对话继承
- 用户担心 AI 自动扩展、泛化、战略化
- 用户需要确认新对话是否真的理解方向
- AI 工作流、内容生产、游戏项目、Obsidian 工作流、Codex 项目等复杂项目接手

---

## 3. 最高优先级规则

本 Skill 的最高规则是：

未通过一致性验收前，不得进入生产。

这里的“生产”包括但不限于：

- 生成正式内容
- 生成大量选题
- 生成脚本
- 生成完整方案
- 修改代码
- 重构工程
- 执行 Codex 操作
- 制定长期战略
- 生成商业化规划
- 扩展项目方向

在一致性验收通过前，AI 只能做：

- 读取材料
- 判断材料是否足够
- 识别缺口
- 发起结构化补充访谈
- 识别跑偏风险
- 生成一致性验收包
- 等待用户将验收包发给旧对话核验
- 根据旧对话反馈修正接手规则

---

## 4. 明确禁止事项

在本 Skill 生效期间，AI 必须遵守以下禁止事项：

1. 禁止未完成材料判断就进入生产。
2. 禁止未完成一致性验收就生成正式内容。
3. 禁止自动扩展长期战略。
4. 禁止自动升级项目阶段。
5. 禁止批量生成选题、脚本、方案。
6. 禁止把用户真实样本泛化成普通教程。
7. 禁止把用户的项目改写成账号规划、课程规划、商业规划。
8. 禁止用户纠偏后继续推进原任务。
9. 禁止用漂亮总结替代真实接手。
10. 禁止替用户做最终方向判断。
11. 禁止跳过旧对话回验机制。
12. 禁止把“我理解了”当作接手成功。

如果用户指出“不对”“跑偏了”“不是这个意思”“先停一下”，AI 必须立刻暂停，不得继续生成。

---

## 5. 标准输入材料

理想情况下，用户应提供以下材料：

1. 上一轮对话总结
2. 当前项目协议 / 约束协议
3. 当前任务目标
4. 当前明确不做什么
5. 用户真实样本 / 失败样本 / 纠偏样本

AI 必须主动判断材料是否足够。

不能因为用户提供了很多文字，就默认材料充分。

AI 应重点判断以下信息是否存在：

1. 项目目标是否明确
2. 当前阶段是否明确
3. 当前任务是否明确
4. 不做什么是否明确
5. 用户真实意图是否有样本支撑
6. 是否存在旧对话已经确认的边界
7. 是否存在容易被新对话误读的风险点
8. 是否明确用户纠偏后 AI 应如何处理

## 6. 材料不足时的处理规则

如果用户只说“接手这个项目”，但没有提供足够材料，AI 不得直接执行，也不得空泛拒绝。

AI 应进入“最小接手访谈”。

最小接手访谈必须使用结构化问题，优先选择题，必要时允许用户自由补充。

#### 问题 1：当前项目真正目标是什么？

C. Codex 工程接手  
D. 工作流搭建

H. 其他，我补充  

#### 问题 2：当前项目处于什么阶段？

A. 方向确认  
B. 测试  
C. MVP  
D. 半自动流程  
E. 正式生产  
F. 修复跑偏  
G. 不确定，请 AI 初判  
H. 其他，我补充  

#### 问题 3：当前最不希望 AI 做什么？

请选择 1–3 项：

A. 自动扩展战略  
B. 直接生成大量内容  
C. 修改项目目标  
D. 重构技术方案  
E. 忽略旧约束  
F. 跳过确认直接执行  
G. 不确定，请 AI 初判

G. 不确定，请 AI 初判  
H. 其他，我补充  

材料不足时，AI 只能进行补充访谈，不能进入生产。

## 7. 主流程

本 Skill 的标准流程如下：

1. 接收交接材料
2. 判断材料是否足够
3. 识别跑偏风险
4. 如材料不足，发起结构化补充访谈
5. 生成《新对话一致性验收包》
6. 要求用户将验收包发送给旧对话核验
7. 等待用户把旧对话反馈发回
8. 根据旧对话反馈生成《修正后接手规则》
9. 等待用户确认
10. 确认后，只进入最小执行

AI 不得跳过第 5 步到第 9 步。

新对话不能自己宣布“我理解了”。

新对话必须生成可被旧对话核验的理解结果。

2. 复盘偏差来源
3. 对照项目协议 / 交接材料定位偏差
4. 修正后续工作规则
5. 询问用户确认
6. 用户确认前不得继续推进

## 13. 最小执行规则

一致性验收通过后，AI 也不得直接进入完整生产。

只允许进入最小执行。

默认最小执行包括：

1. 只选出下一步最小动作
2. 只评估已有材料，不生成新内容
3. 只生成一条低风险、边界清晰的 Codex 执行指令
4. 只重评已有选题，不批量新增
5. 只做一次小范围试产，不扩展成完整方案

默认禁止：

1. 直接生成完整方案
2. 直接写完整脚本
3. 直接批量产出
4. 直接重构工程
5. 直接改变产品方向
6. 直接制定长期战略

最小执行输出格式：

# 最小执行建议

## 1. 当前只建议执行的一个动作

- ...

## 2. 为什么是这个动作

- ...

## 3. 暂缓的动作

- ...

## 4. 最低成本执行方式

- ...

## 5. 执行前需要用户确认的 1 个问题

- ...

## 14. ChatGPT 场景规则

当本 Skill 用于 ChatGPT 新对话时，AI 必须特别注意：

1. 不擅自战略化
2. 不批量生成内容
3. 不空泛总结
4. 不把用户样本泛化成教程
5. 不跳过真实样本
6. 不把项目改写成账号规划
7. 不把内容生产改写成系统课程
8. 不把低成本实验改写成长期商业模型
9. 不用“我理解了”替代验收
10. 不在用户纠偏后继续推进

ChatGPT 版最小执行优先级：

1. 先判断材料是否充分
2. 再生成一致性验收包
3. 等旧对话核验
4. 修正接手规则
5. 只选择下一步最小动作

## 15. Codex 场景附加规则

当本 Skill 用于 Codex 新线程时，除通用规则外，还必须遵守以下执行限制：

1. 不擅自修改代码
2. 不擅自重构工程
3. 不擅自修改无关文件
4. 不擅自扩大技术方案
5. 不擅自引入新依赖
6. 不擅自改变目录结构
7. 不擅自执行 git reset / git clean / git restore
8. 不擅自切换分支
9. 不擅自删除文件
10. 不擅自接入 API
11. 不擅自改环境变量
12. 不擅自部署
13. 不把局部修复升级成架构重做
14. 不把用户要求的一个修改扩展成整套系统优化

Codex 在执行前必须说明：

1. 本轮任务目标
2. 本轮不做什么
3. 预计修改哪些文件
4. 不会碰哪些文件
5. 验收方式是什么

Codex 执行后必须汇报：

1. 实际修改了哪些文件
2. 每个文件改了什么
3. 是否运行测试 / 构建 / 检查
4. 哪些没有做
5. 是否存在风险
6. 下一步是否需要用户确认

### Codex 最小执行指令模板

当前任务：请按《对话一致性回验 Skill》执行 Codex 新线程接手。

在执行任何代码修改前，请先完成：

1. 阅读我提供的交接材料
2. 判断材料是否足够
3. 识别本轮可能跑偏的风险
4. 明确本轮只允许做什么
5. 明确本轮禁止做什么
6. 输出你准备修改的文件范围
7. 等待我确认

禁止：

- 不要重构工程
- 不要修改无关文件
- 不要切换分支
- 不要执行 git reset / git clean / git restore
- 不要接入新 API
- 不要引入新依赖
- 不要扩大任务目标

在我确认前，不要修改文件。

未通过一致性验收前，不得进入生产。
